# SAGP Architecture Manifesto (Phase II)

## Purpose

The Society for the Advancement of Geohistory and Philosophy (SAGP) Executive Portal is not a collection of independent applications. It is a knowledge platform built around a small number of stable abstractions. Every component of the system should reinforce these abstractions rather than introducing new ones.

The architecture is organized into six layers.

---

# Layer 0 — Canonical Knowledge

Canonical knowledge consists of the authoritative sources of truth maintained by the Society.

Examples include:

- Membership database
- Canonical publication YAML
- Governance documents
- Conference information
- Election records
- Financial records

Canonical knowledge is never generated by downstream systems. It is the foundation from which all other artifacts are derived.

---

# Layer 1 — Ontology (`sagp_core`)

The ontology defines the fundamental objects that exist within the SAGP digital ecosystem.

Examples include:

- Member
- Audience
- Publication
- Message
- Communication
- Conference
- Committee
- Motion
- Resolution
- Election
- Officer

These objects represent the language of the Society.

They are independent of user interfaces, databases, programming languages, file formats, and implementation details.

Applications manipulate these objects; they do not define them.

---

# Layer 2 — Capabilities (Contracts)

Capabilities define the transformations that the platform promises to perform.

Examples include:

- Generate an `Audience` from canonical membership knowledge.
- Generate a `Message` from canonical publication knowledge.
- Compose a `Communication` from an `Audience` and a `Message`.

Capabilities specify **what** must be accomplished, but never prescribe **how** it is implemented.

Every engine is an implementation of one or more capability contracts.

---

# Layer 3 — Engines

Engines are deterministic implementations of capabilities.

Examples include:

- Membership Engine
- Publishing Engine
- Communication Engine
- Governance Engine
- Conference Engine (future)

An engine transforms canonical knowledge into one or more derived objects.

Given identical inputs, an engine should produce identical outputs.

Whenever practical, engines should be stateless.

---

# Layer 4 — Workspaces

Workspaces provide the human interface to the platform.

Examples include:

- Membership Workspace
- Publishing Workspace
- Communications Workspace
- Governance Workspace
- Administration Workspace

A workspace may orchestrate multiple engines.

Business logic belongs in engines.

User interaction belongs in workspaces.

---

# Layer 5 — Workflows

Workflows compose capabilities into meaningful organizational processes.

Example:

    Generate Audience
            ↓
    Generate Message
            ↓
    Compose Communication
            ↓
        Review
            ↓
        Approve
            ↓
        Deliver
            ↓
        Archive

Workflows describe how the Society operates rather than how software is internally organized.

---

# Core Architectural Principles

## 1. Canonical Knowledge

There is exactly one authoritative source for each category of institutional knowledge.

Everything else is derived.

---

## 2. Derived Objects

Derived objects are reproducible products of deterministic transformations.

Examples include:

- Audience
- Message
- Communication

Derived objects may be archived, versioned, audited, compared, regenerated, and reused.

---

## 3. Separation of Concerns

Each architectural layer has a single responsibility.

- Canonical Knowledge stores truth.
- Ontology defines the objects.
- Capabilities define behavior.
- Engines implement behavior.
- Workspaces support users.
- Workflows coordinate organizational processes.

---

## 4. Communication Algebra

Communication is defined by the composition

    Audience ⊕ Message → Communication

where

- Audience is produced by the Membership Engine.
- Message is produced by the Publishing Engine.
- Communication is produced by the Communication Engine.

A Communication is a first-class derived object containing:

- recipients
- subject
- rich HTML
- plain text
- delivery metadata
- audit metadata

---

## 5. Artifact-Oriented Architecture

Every derived object may exist as a durable artifact independent of any running application.

Examples include:

- Audience
- Message
- Communication

Artifacts support:

- reproducibility
- interoperability
- auditing
- archival
- automation

Representations such as CSV files, HTML pages, email messages, clipboard contents, or PDFs are exports of these artifacts—not the artifacts themselves.

---

## 6. Stable Dependency Direction

Dependencies always flow downward.

    Canonical Knowledge
            ↓
    Ontology (sagp_core)
            ↓
    Capabilities (Contracts)
            ↓
        Engines
            ↓
      Workspaces
            ↓
       Workflows

Lower layers never depend upon higher layers.

This preserves modularity, testability, maintainability, and long-term architectural stability.

---

# Architectural Philosophy

The Executive Portal is not the system.

It is the orchestration environment through which Society officers interact with the Society's knowledge ecosystem.

The platform is designed around stable semantic objects rather than applications.

Applications may evolve.

User interfaces may change.

Storage technologies may be replaced.

Delivery mechanisms may come and go.

The ontology and the contracts should remain stable.

This separation allows the platform to evolve organically while preserving conceptual integrity.

---

# Long-Term Vision

The architecture is intended to scale by introducing new ontology objects, capabilities, engines, workspaces, and workflows without disrupting existing components.

Future capabilities may include:

- Conference management
- Governance support
- Elections
- Financial reporting
- Analytics
- AI-assisted decision support
- Strategic forecasting

Each new capability should integrate naturally into the existing architectural layers rather than introducing parallel structures.

The long-term objective is not simply to build software, but to construct a coherent, extensible, constitution-governed knowledge platform that models how the Society operates and provides a stable foundation for future human and AI collaboration.
